## Technical Debt

### TD-1: Bare Exception Handlers
- **Location:** Various claim modules in `backend/progress/`
- **Issue:** Some modules use broad `except Exception` blocks that catch everything, making debugging harder
- **Risk:** Low — claims are deterministic and well-tested, but error messages may be unhelpful

### TD-2: Counter Maintenance Burden
- **Location:** `index.html` (11 places), `README.md`, `AGENTS.md`, `llms.txt`, `system_manifest.json`, `CONTEXT_SNAPSHOT.md`, `known_faults.yaml`
- **Issue:** Claim count, test count, and other numbers must be updated in 7+ files (11 locations in index.html alone)

- **Mitigation:** CLAUDE.md documents all locations; PowerShell batch replace for index.html

### TD-3: Stale Counter in known_faults.yaml

## Security Considerations

### SEC-1: No Input Size Limits
- **Location:** `backend/progress/runner.py`, claim modules
- **Issue:** No payload size validation before processing — large inputs could consume memory
- **Risk:** Low in current use (single-user CLI tool), but relevant if exposed as a service

### SEC-2: HMAC-SHA256 vs Ed25519
- **Location:** `scripts/mg_sign.py`
- **Issue:** Bundle signing uses symmetric HMAC-SHA256 (shared secret). Ed25519 asymmetric signing is planned but not yet implemented.
- **Risk:** Low — HMAC is cryptographically sound, but shared secret model limits multi-party scenarios
- **Planned:** Ed25519 upgrade is on the roadmap

### SEC-3: File Path Handling
- **Location:** `backend/progress/data_integrity.py`, `scripts/mg.py`
- **Issue:** `fingerprint_file()` normalizes CRLF→LF for cross-platform consistency, but path traversal is not explicitly guarded
- **Risk:** Low — CLI tool, not a web service

## Performance Considerations

### PERF-1: Linear Ledger Scans
- **Location:** `backend/ledger/ledger_store.py`
- **Issue:** Ledger is append-only JSONL — queries scan linearly. Currently ~75 entries.
- **Risk:** Low now, but will degrade with thousands of entries
- **Mitigation:** Index or SQLite migration if ledger grows significantly

### PERF-2: Sequential Claim Execution
- **Location:** `backend/progress/runner.py`
- **Issue:** All 14 claims execute sequentially — no parallel execution
- **Risk:** Low — each claim runs in <1s. Would matter at scale (100+ claims)

### PERF-3: JSON Serialization
- **Location:** Throughout — all artifacts are JSON
- **Issue:** Large datasets serialized to JSON may be slow
- **Risk:** Low — current datasets are small (CSV fixtures < 100KB)

## Fragile Areas

### FRAG-1: Runner Dispatch
- **Location:** `backend/progress/runner.py`
- **Issue:** Adding a new claim requires a manual dispatch block — no auto-discovery
- **Risk:** Medium — forgetting to add dispatch block means claim silently doesn't run

### FRAG-2: Pack Manifest Integrity
- **Location:** `scripts/mg.py`
- **Issue:** Verification depends on exact file list in manifest matching pack contents. Any extra/missing file breaks Layer 1.
- **Risk:** Low — by design (this IS the integrity guarantee)

### FRAG-3: Cross-Platform Path Handling
- **Location:** `backend/progress/data_integrity.py`
- **Issue:** CRLF→LF normalization handles the main cross-platform issue, but path separators in manifests could differ between Windows/Unix
- **Risk:** Low — tests pass on Windows (current dev environment)
