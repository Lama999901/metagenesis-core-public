# Architecture

**Analysis Date:** 2026-03-16

## Pattern Overview

**Overall:** Layered verification pipeline with claim-driven evidence packaging and cryptographic proof-chaining.

**Key Characteristics:**
- Three independent verification layers (integrity, semantic, step-chain) that progressively catch sophisticated tampering
- Claim-based architecture: 14 domain-specific computational claims (MTR-1/2/3, ML_BENCH-01/02/03, DATA-PIPE-01, SYSID-01, DRIFT-01, DT-FEM-01, DT-SENSOR-01, DT-CALIB-LOOP-01, PHARMA-01, FINRISK-01)
- Execution trace with deterministic hash-chaining over computation steps (Layer 3)
- Append-only ledger integration for traceability (job creation, execution, completion events)
- Physical anchor principle: verification grounded in physical constants (e.g., E = 70 GPa for MTR-1), not arbitrary thresholds

## Layers

**Layer 1 — SHA-256 Integrity:**
- Purpose: Detect file-level modifications after packaging
- Location: `scripts/mg.py` → `_verify_pack()` function
- Contains: Manifest file hash validation, root_hash reconstruction
- Depends on: `pack_manifest.json` (generated by `scripts/steward_submission_pack.py`)
- Used by: `mg verify --pack` CLI command

**Layer 2 — Semantic Verification:**
- Purpose: Detect evidence stripping, bundle composition attacks, cross-domain mismatches
- Location: `scripts/mg.py` → `_verify_semantic()` function (lines 128-248)
- Contains: Evidence structure validation, job_kind matching, payload verification, uncertainty bounds checking, anchor_hash format validation
- Depends on: `evidence_index.json` (index mapping claim_id → normal/canary bundles with artifact paths)
- Used by: `mg verify --pack` CLI command

**Layer 3 — Step Chain Verification:**
- Purpose: Detect computation input manipulation, step reordering, intermediate result tampering
- Location: `scripts/mg.py` → embedded in `_verify_semantic()` (lines 204-235)
- Contains: Execution trace structure validation, hash continuity checks (final step hash must equal `trace_root_hash`)
- Depends on: `execution_trace` list in run artifacts + `trace_root_hash` claim
- Used by: Semantic verification layer

**Layer 4 — Bundle Signing (Post-PPA Innovation #6):**
- Purpose: Verify bundle creator identity and timestamp
- Location: `scripts/mg_sign.py` (HMAC-SHA256 signing)
- Contains: Bundle metadata signing, verification key management
- Depends on: Layers 1-3 pass before signing
- Used by: `mg sign-pack` CLI command (optional, for attribution)

## Data Flow

**Claim Execution Flow (Normal or Canary Mode):**

1. **Job Creation**: `ProgressRunner.create_job(payload)` in `backend/progress/runner.py`
   - Creates Job with job_id, trace_id
   - Emits LedgerEntry (phase 31, actor "scheduler_v1", action "job_created")
   - Stores in JobStore

2. **Job Execution**: `ProgressRunner.run_job(job_id, canary_mode)`
   - Updates job status → RUNNING, records started_at
   - Calls `_execute_job_logic(job)` which dispatches to claim-specific implementation
   - For MTR-1: invokes `backend/progress/mtr1_calibration.run_calibration()`
   - Claim handler computes result + builds 4-step execution trace
   - Updates job status → SUCCEEDED, records completed_at
   - Emits LedgerEntry (phase 31, action "job_completed" or "job_completed_canary")

3. **Artifact Emission**:
   - Run artifact: JSON file under `reports/progress_runs/` containing trace_id, job_snapshot, execution_trace, trace_root_hash, canary_mode flag
   - Ledger snapshot: JSONL file under `reports/ledger_snapshots/trace_<trace_id>.jsonl` with all ledger entries for that trace

4. **Evidence Indexing**: `backend/progress/evidence_index.py`
   - Scans progress_runs/*.json and ledger_snapshots/*.jsonl
   - Returns dict: trace_id → list of artifacts (path, type, size)
   - Used to build evidence_index.json for packing

5. **Pack Building**: `scripts/steward_submission_pack.py`
   - Reads scientific_claim_index.md, canonical_state.md, phase_registry_v0_1.md
   - Collects evidence artifacts (normal + canary runs)
   - Generates pack_manifest.json (Layer 1: file-level SHA-256s + root_hash)
   - Generates evidence_index.json (Layer 2: semantic structure)
   - Writes submission pack with overview document

6. **Pack Verification**: `scripts/mg.py → cmd_verify_pack()`
   - Layer 1: Reconstruct root_hash from pack_manifest.json
   - Layer 2: Semantic validation via evidence_index.json
   - Layer 3 (embedded in Layer 2): Step chain validation
   - Returns PASS/FAIL with detailed report JSON

**Physical Anchor Chain (Cross-Claim Dependency):**

```
MTR-1 execution
  ↓ (computes E_hat, trace_root_hash, anchor_hash = final step hash)
  ↓ bakes anchor_hash into DT-FEM-01 Step 1 inputs
DT-FEM-01 execution
  ↓ (FEM solve uses MTR-1 anchor, computes displacement, trace_root_hash)
  ↓ bakes anchor_hash into DRIFT-01 Step 1 inputs
DRIFT-01 execution
  ↓ (compares current E against MTR-1 anchor)
  ↓ trace_root_hash cryptographically commits entire chain
```

Verify end-to-end: `mg verify-chain bundle_mtr1/ bundle_dtfem/ bundle_drift/`

**State Management:**

- Job state: JobStatus enum (QUEUED, RUNNING, SUCCEEDED, FAILED) in `backend/progress/models.py`
- Persistence: No database; artifacts are immutable JSON files + JSONL ledger
- Ledger storage: Append-only JSONL with optional fcntl file locking (Unix) in `backend/ledger/ledger_store.py`
- In-memory stores: JobStore and LedgerStore initialized per runner instance (deterministic for testing)

## Key Abstractions

**Job:**
- Purpose: Represents a computational claim execution with full traceability
- Examples: `backend/progress/models.py::Job` dataclass
- Pattern: Dataclass with UUID-based job_id and trace_id, ISO8601 timestamps, status lifecycle

**LedgerEntry:**
- Purpose: Immutable record of job event (creation, execution, completion) for non-repudiation
- Examples: `backend/ledger/models.py::LedgerEntry` dataclass
- Pattern: Append-only entry with phase, actor, action, inputs/outputs, artifacts, legal_sig_refs

**Run Artifact (JSON):**
- Purpose: Complete evidence snapshot for one claim execution
- Structure: trace_id, job_snapshot (payload + result), execution_trace (4-step hash chain), trace_root_hash, canary_mode, timestamps
- Example file: `reports/progress_runs/mtr1_<trace_id>.json`
- Pattern: Deterministic JSON serialization (sorted keys) to ensure reproducibility

**Execution Trace (4-step chain):**
- Purpose: Cryptographic proof of computation step sequence and input integrity
- Pattern:
  ```
  Step 1: h1 = SHA256(init_params + "genesis")
  Step 2: h2 = SHA256(step2_data + h1)
  Step 3: h3 = SHA256(step3_data + h2)
  Step 4: trace_root_hash = SHA256(step4_data + h3)
  ```
- Used by: Every claim implementation (MTR-1, MTR-2, MTR-3, etc.)
- Verified by: Layer 3 checks final step hash against claimed trace_root_hash

**Evidence Index (JSON):**
- Purpose: Semantic registry linking claims to artifact paths (run + ledger), job_kind, normal/canary modes
- Structure: claim_id → {job_kind, normal: {run_relpath, ledger_relpath}, canary: {...}}
- Example: `evidence_index.json` in submission pack
- Pattern: Built by evidence_index.py from progress_runs and ledger_snapshots scans

**Pack Manifest (JSON):**
- Purpose: Layer 1 integrity proof: list of files with SHA-256 + root_hash
- Structure: {files: [{relpath, sha256}, ...], root_hash: "..."}
- Example: `pack_manifest.json` in submission pack
- Pattern: Generated deterministically by steward_submission_pack.py

## Entry Points

**CLI Entry Point:**
- Location: `scripts/mg.py`
- Triggers: User runs `python scripts/mg.py <subcommand> <args>`
- Responsibilities: Argument parsing, subcommand dispatch (steward-audit, pack-build, verify, verify-chain, sign-pack), error handling

**Claim Execution Entry Point:**
- Location: `backend/progress/runner.py::ProgressRunner.run_job()`
- Triggers: Test suite or demo calls runner.run_job(job_id, canary_mode)
- Responsibilities: Job lookup, status transitions, ledger event emission, dispatch to claim-specific logic

**Job Dispatch (Claim Kind → Implementation):**
- Location: `backend/progress/runner.py::_execute_job_logic()` (conceptual; actual dispatch depends on payload["kind"])
- Triggers: ProgressRunner.run_job() when job status is RUNNING
- Responsibilities: Route to correct claim handler (mtr1_calibration.run_calibration, etc.), collect result, emit trace

## Error Handling

**Strategy:** Explicit validation at layer boundaries; fail-fast with specific error messages.

**Patterns:**

- **Layer 1 (Integrity)**: File-level validation—missing manifest, invalid JSON, hash mismatch → returns (False, error_msg, report_dict)
- **Layer 2 (Semantic)**: Structure validation—missing required keys, payload mismatch, invalid hashes → specific error message + check list
- **Layer 3 (Step Chain)**: Hash continuity—execution_trace missing, step hash invalid, trace_root_hash broken → msg includes artifact path and step number
- **Job Execution**: JobStore.get() returns None if not found; JobStatus checks prevent rerunning; LedgerEntry validation at append time (SHA-256 format, ISO8601 timestamp)

**Example** (`scripts/mg.py:_verify_semantic` lines 162-166):
```python
for key in ("trace_id", "job_snapshot", "canary_mode"):
    if key not in art:
        msg = f"Run artifact {run_rel} missing required key: {key}"
        return False, msg, [msg]
```

## Cross-Cutting Concerns

**Logging:**
- Framework: Python stdlib logging
- Approach: Structured logging at key points (job creation, execution start/end, ledger writes, artifact emission)
- Examples: `backend/progress/runner.py` ProgressRunner.__init__(), runner.run_job()
- Pattern: logger.info() for state transitions, logger.warning() for non-fatal issues

**Validation:**
- Framework: Inline validation functions + dataclass __post_init__ hooks
- Approach: Layer-boundary validation (manifest structure, semantic evidence, execution trace)
- Examples: `backend/ledger/models.py::validate_sha256()`, `validate_iso8601()`, LedgerEntry.__post_init__()
- Pattern: Raise ValueError with specific message on validation failure

**Authentication/Attribution:**
- Approach: Ledger entries embed actor and legal_sig_refs (e.g., "metagenesis-core-ppa-63996819")
- Examples: `backend/progress/runner.py` emits actor="scheduler_v1" (normal) or "scheduler_v1_canary" (canary)
- Pattern: Layer 4 (bundle signing) adds HMAC signature over pack_manifest.json for creator verification

**Reproducibility:**
- Approach: Deterministic JSON serialization (sorted_keys=True, separators=(',',':')), seeded RNG in claim handlers
- Examples: `backend/ledger/ledger_store.py::append()` uses json.dumps(..., sort_keys=True)
- Pattern: Test suite normalizes artifacts (strips timestamps, trace_id) to compare computation results
